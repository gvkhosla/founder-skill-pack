Brutal truth

Founder Skills today

  • Feels like: an unusually good founder skill pack.
  • Weakness: not yet stateful, not fully orchestrated, limited eval/testing, repo drift.
  • Risk: stays “excellent content” instead of becoming product infrastructure.

Founder Skills OS target

  • Best at: deciding what matters next across the whole company.
  • Feels like: a founder operating system with memory and specialist workflows.
  • Has: orchestration, state, handoffs, readiness gates, tests/evals, connectors.
  • Moat: it compounds company knowledge over time.
Do not add 50 disconnected skills. Build one stateful system where every skill is a node in the company graph.

What to steal from gstack

Steal the system ideas, not just the commands

1. Workflow chaining

Every output feeds another workflow. Founder Skills should chain product → engineering → GTM → sales → ads → support → hiring → weekly review.

2. Central orchestrator

Upgrade founder-partner into a real router that reads state, sees bottlenecks, and invokes the right specialist automatically.

3. Packaging discipline

Generated docs, host-specific packaging, install/update flows, and one-command onboarding.

4. Gates + readiness

Launch gates, PMF gates, sales readiness, ads readiness, hiring readiness — pass/fail, not vague encouragement.

5. Tests + evals

Scenario evals, regression tests, cross-skill consistency, output quality scoring.

6. “System feeling”

The user should feel like they are entering an operating environment, not just picking a markdown template.

New product architecture

Founder Skills OS should have five layers

Layer 1

Specialists

Skills across engineering + product, GTM, sales, SEO/GEO, ads, support, hiring, ops, and execution.

Layer 2

Orchestrator

Founder-partner 2.0 reads state, identifies the bottleneck, chooses next move, and updates memory.

Layer 3

State engine

Structured company state: stage, metrics, goals, PMF, current bottleneck, active experiments, confidence, freshness.

Layer 4

Artifact graph

Every artifact knows what it depends on, what it enables, and whether it is stale or superseded.

Layer 5

Execution handoffs

Send decisions into implementation, repos, GTM systems, CRM, ad tools, docs, QA, deploy, and test flows.

Canonical state objects

  • company-state.json
  • founder-context.md (human-readable layer)
  • artifacts/index.json
  • experiments/ + status

Core metadata on every artifact

  • created_by
  • depends_on
  • feeds_into
  • confidence
  • freshness
  • recommended_next_skill
Company graph

Artifacts should become a living company graph, not a folder of isolated markdown files

Example node: positioning.md

  • depends on: customer profile, problem validation
  • feeds into: landing page, outbound, ads, launch plan
  • freshness: stale if customer profile changed
  • confidence: medium until tested in sales calls and ads
  • next move: run landing-page-copywriter or outbound-sequence writer

Why this matters

  • Prevents drift between strategy and execution.
  • Makes the system cumulative over months, not just per session.
  • Lets the orchestrator explain why a next step is recommended.
  • Creates a data model for UI, tests, and collaboration.

Tests and evals

If you want this to be serious, test the company workflows — not just the code

Test types to build

  • Skill structure tests — frontmatter, sections, outputs, related links.
  • Cross-skill consistency tests — README/site/llms/docs all match source of truth.
  • Scenario evals — 20–50 canonical startup cases.
  • Output quality scoring — specificity, decisiveness, usefulness, realism.
  • Regression tests — skill improvements do not make outputs more generic.
  • End-to-end workflow tests — full company sequences still cohere.
  • Build workflow tests — product plan → implementation plan → review → QA → release readiness.

Scenarios to benchmark

  • solo founder pre-idea
  • nontechnical founder with agency
  • B2B SaaS with demos but no closes
  • consumer app with activation drop-off
  • product with traffic but weak conversion
  • revenue but no repeatable growth channel
  • ads spend rising, CAC worsening
  • team unsure whether to hire sales or growth first
  • founder has strong demand clarity but no build confidence
  • founder shipped v1 but quality/release process is weak
The bar: the system should not just sound smart. It should produce artifacts that a founder can use to change behavior within 7 days.

Bottom line

The positioning statement

gstack helps one person behave like a high-output software team.

Founder Skills OS should help one founder behave like a high-judgment company builder with integrated build muscle.

One-line vision

Founder Skills OS is the operating system for AI-native startups: it turns ambiguity into decisions, decisions into buildable and sellable artifacts, artifacts into execution, and outcomes back into company memory.

Success criteria

  • The founder knows the top bottleneck this week.
  • The system knows what changed since last week.
  • The next skill or sequence is obvious.
  • Outputs are specific enough to build, launch, and sell immediately.
  • The whole system gets smarter as the company evolves.
