What gstack is

What gstack actually does

gstack is not just a set of prompts. It is a workflow system for AI-assisted software delivery: slash-command skills, generated host-specific docs, browser tooling, safety utilities, routing rules, update/config logic, and test/eval infrastructure.

Why gstack feels powerful

gstack wins on systemization, integration, and proof

What makes it powerful

  • Connected workflow — each step feeds the next.
  • Operational maturity — config, updates, telemetry, learnings, routing rules.
  • Browser + QA — gives the agent eyes and a verification loop.
  • Cross-host packaging — Claude, Codex, OpenCode, Cursor, Factory, Slate, Kiro, OpenClaw.
  • Tests/evals — much more software discipline around the skill system.

Important nuance

gstack is strongest when the problem is: how do I ship software fast, with review, browser testing, and deploy discipline? It is less unique on PMF thinking, founder memory, and ongoing company learning loops.

Current issues to fix

Before going beyond gstack, Founder Skills needs a stronger core

Repo / product drift found

  • Repo contains 27 skills, but parts of tooling/docs still say 24.
  • scale exists in content/site, but CLI + validator still lag it.
  • npm run check currently fails.
  • Several skills violate the repo's own authoring standard.
  • No tests/evals around output quality.

Strategic weakness

  • Today the pack feels like excellent content with lighter tooling.
  • The site is elegant but under-proves impact.
  • Artifacts are mostly static files, not a compounding system.
  • Skills are isolated; there is not yet a strong state engine connecting them.

Rule

Do not add more skills first. First make the existing pack truthful, testable, linked, and stateful.

How to go beyond

Six moves that make Founder Skills significantly more powerful

1. Add a state engine

Create a canonical company state: stage, goals, north star, PMF signal, active experiments, open questions, founder type, current bottleneck.

2. Turn artifacts into a graph

Every artifact should know what it depends on, what supersedes it, confidence score, freshness, and what skill should run next.

3. Add evidence connectors

Import analytics, Stripe, churn data, interview notes, support tickets, launch metrics, repo activity, and experiments into the skills.

4. Ship sequences, not just single skills

Pre-built plays: 0→1 validation, launch prep, PMF diagnosis, post-launch recovery, first hire, weekly founder review.

5. Add evals and benchmarks

Test skills on real founder scenarios. Score artifact specificity, decisiveness, usefulness, and business impact.

6. Build execution handoffs

One-click handoff from artifact to build tool: send positioning to landing-page builder, MVP brief to implementation agent, launch plan to GTM agent.

Concrete roadmap

90-day plan to make Founder Skills meaningfully stronger

Days 0–30

Fix truth + build the foundation

  • One source of truth for phases, skill count, outputs.
  • Generate README, llms.txt, site cards, install docs from source.
  • Update validator for scale and 27 skills.
  • Add skill linting: section checks, output checks, related-skill checks.
  • Choose 3 flagship skills and make them exceptional.
Days 30–60

Add orchestration + company state

  • Ship company-state.json or structured founder-context schema.
  • Upgrade founder-partner into the main router.
  • Add “recommended next skill” logic everywhere.
  • Create 3 sequences: validation, launch, PMF diagnosis.
  • Build a simple dashboard UI over artifacts and state.
Days 60–90

Add compounding advantage

  • Add connectors for analytics, Stripe, interview notes, support feedback.
  • Build eval scenarios with before/after artifact scoring.
  • Add handoff outputs for pi / gstack / Codex / Claude execution.
  • Publish founder case studies with proof of decisions and outcomes.
  • Optionally add premium benchmark datasets later.
What to build first

If you want one ruthless priority stack

Build these first

  1. Source-of-truth generator for skills/site/docs/install.
  2. Founder-partner 2.0 as the default entrypoint and router.
  3. Artifact graph + freshness/confidence metadata.
  4. Three canonical sequences instead of more standalone skills.
  5. Eval harness using 20 real founder scenarios.

Do not build yet

  1. Not a browser/QA stack to compete directly with gstack.
  2. Not dozens of new skills before the existing system is coherent.
  3. Not fancy packaging before you have state + orchestration.
  4. Not vague “AI founder” branding without hard proof and examples.
The winning Founder Skills product is not “more founder prompts.” It is a live company operating system that remembers, scores, routes, and tells the founder what matters next.
Bottom line

The simplest strategy statement

gstack helps a builder behave like a high-output software team.

Founder Skills should help a founder behave like a high-judgment company builder.

One-line positioning

Founder Skills is the founder operating system for AI-native startups: it turns ambiguity into decisions, decisions into artifacts, artifacts into execution handoffs, and outcomes back into memory.

Success test

  • The founder starts with confusion and leaves with a sharper weekly focus.
  • The system knows what changed since last week.
  • The next recommended move is obvious.
  • The handoff to execution tools is immediate and high-quality.
